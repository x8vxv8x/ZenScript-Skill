## 规则
1. **检查该mod是否已经存在一份API文档。** 如果目录中存在此mod的API文档，则在此文档的基础上进行查漏补缺，补充没有记录的方法。
2. **只记录文档中明确存在的方法。** 不要猜测或推断未文档化的方法。
3. **API 名称必须与文档完全一致。** 包括大小写。
4. **参数类型要准确。** 使用 ZenScript 类型名（IItemStack、IIngredient、string、int、float、bool、IData 等）。
5. **每个方法都要有说明。** 即使看起来很明显。
6. **如果文档中有多个版本的 API（如简化版和完整版），全部列出。**
7. **如果某些信息文档中没有，标注"未说明"而不是猜测。** 比如方法的返回类型，参考文档没有明确写出时标注"未说明"。
8. **使用中文编写说明和注释，API 名保持英文。**
9. **如果一个模组有多个 ZenScript 类（如不同机器的管理器），每个类单独一个章节。**
10. **如果用户提供的资料中对方法中的参数有定义或解释，请添加到方法说明中，如果没有，直接不写**
11. **对于属性，如果只读，分到@ZenGetter，如果可读写，分到@ZenGetter / @ZenSetter** 严格执行属性分类，不要把只读属性放到@ZenGetter/@ZenSetter中，也不要把可读写属性放到@ZenGetter中。
12. **使用示例，常见错误和注意事项必须来自用户提供的资料，如果没有，那么就不写，绝对不能自己编造**
13. **在vanilla或utils内创建新文件后请更新 `references\index.md` 索引文件**

## 文件组织规则

### 归类原则

1. **按被扩展的模组归类**，而不是按提供 API 的模组归类。例如 zenutils 提供的 ContentTweaker 扩展（`mods.zenutils.cotx.*`）应放入 `references/mods/contenttweaker.md`，而不是留在 `zenutils.md` 中
2. **扩展原版类型（IEntity、IPlayer、IWorld 等）的 API** → 移到 `references/vanilla/` 下对应的功能文件中，不需要新增文件，用户已经分类好了
3. **每个模组提供的方法集中在一起**，在该组方法前用一个统一的 import 块标注来源和导入语句

### 标注格式

一个文件中包含来自不同模组的 API 时，使用以下格式分组：

```markdown
---

## [来源模组名] 扩展（需安装 [来源模组名]）

> `import mods.source_modid.ClassName1;`
> `import mods.source_modid.ClassName2;`

### [API 类名]
- 这里填写API列表，格式与上方输出要求相同。

```
